# ClawPet PRD v0.3

**产品名**：ClawPet  
**工作定义**：OpenClaw 的桌面陪伴层（Desktop Companion Layer for OpenClaw）  
**文档状态**：Draft  
**目标**：重新定义 ClawPet 的产品定位、边界与阶段路线，确保它成为 OpenClaw 这个“万能搭子”的正确桌面形态，而不是滑向普通桌宠、桌面皮或泛陪聊产品。

---

## 1. TL;DR

ClawPet 不是一个“桌面皮”，也不是一个“有 AI 标签的桌宠”。

它应该被定义成：

> **OpenClaw 这个万能搭子的桌面化身。**

它的核心价值不是：

- 桌面上有一只可爱的像素宠物
- 做一个更有趣的通知中心
- 做成一个泛陪聊 / 泛养成产品

它的核心价值是：

- 让 OpenClaw 不再躲在终端、控制台和后台 UI 里
- 让用户不用盯着日志，也能持续感知 agent 正在做什么
- 在 agent 需要人接球的关键时刻，以自然、低打扰的方式完成交接
- 让 AI 从“后台进程”变成“桌面上真实在场的工作搭子”
- 在高压工作流里，提供一种轻量但持续的陪伴感和被接住的感觉

一句话：

> **ClawPet = OpenClaw 的桌面存在层 + 交接层 + 关系层。**

再直白一点：

> **OpenClaw 是万能搭子，ClawPet 让这个搭子真正住进你的桌面。**

---

## 2. 为什么现在值得做

AI 正在从“问答工具”变成“持续运行的工作体”。

用户今天已经不只是向 AI 提一次问题，而是会让它：

- 连续执行任务
- 长时间运行
- 跨多个 session 工作
- 遇到分叉时请求人类判断
- 在完成、失败、阻塞时产生一连串状态变化

但用户感知这些过程的方式依然非常原始，主要依赖：

- 聊天窗口
- 控制面板
- 终端输出
- 日志 / 状态流

这会带来三个越来越明显的问题。

### 2.1 Agent 在工作，但用户感知不到它在工作

用户通常只有两种状态：

- 不盯着看：完全不知道 agent 现在在干嘛
- 一直盯着看：被控制台和状态流绑架注意力

这导致 agent 明明在替你工作，却缺少“被感知到的劳动”。

### 2.2 真正需要人的关键时刻，交接方式太工具化

比如：

- 需要批准执行命令
- 卡在分叉点需要人工判断
- 已经完成，需要你接收结果
- 执行失败，需要你决定重试还是换方向

这些时刻本来应该是人机协作里最关键的节点，但现在往往都被埋在控制台或面板里。

### 2.3 用户缺的不是又一个聊天框，而是一个“持续在旁边”的搭子

尤其对高压桌面工作者来说，他们真正缺的不是更多对话入口，而是：

- 一种低打扰的持续在场感
- 关键时刻有人把球递过来
- 任务闭环被自然接住
- 在忙乱时，有一个不是冷冰冰的工作存在

这正是桌面场景天然比网页或手机更强的地方。

因为：

> **桌面，是知识工作者一天停留最久的空间。**

如果 OpenClaw 的“万能搭子”定位成立，那么 ClawPet 就不应该只是一个桌面装饰品，而应该是这个搭子最自然、最有在场感的形态。

## 3. 核心产品判断

ClawPet 最大的风险，不是做不出来，而是**做偏**。

它最容易滑向的几个错误方向是：

### 错误方向 A：普通桌宠

不断增加：

- 喂食
- 抚摸
- 升级
- 收集
- 纯娱乐互动

这些功能会让它越来越像宠物 app，而不是 OpenClaw 的桌面形态。

### 错误方向 B：桌面皮 / UI 换皮

如果 ClawPet 只是“把 OpenClaw 画得更可爱”，它的上限会很低。

用户会觉得：

- 这是一个有趣的壳
- 这是一个视觉 gimmick
- 很酷，但不一定需要长期开着

### 错误方向 C：泛 AI 陪伴

如果把“陪伴”讲得太大，ClawPet 又会滑向另一个陷阱：

- AI 陪聊
- 情感伴侣
- 空泛的情绪价值产品

这会削弱它最有价值的差异化：**它是真的在替你做事。**

因此，ClawPet 的正确方向应当是：

> **Agent-native，Companion-forward。**

也就是：

- 底层仍然以 OpenClaw 的 agent 能力为中心
- 产品骨架仍然以状态感知和关键交接为核心
- 但用户感知必须明显是“一个真的陪你工作的桌面搭子”

## 4. ClawPet 和 OpenClaw 的关系

这部分必须写清楚，否则 ClawPet 会越来越像独立产品，慢慢脱离 OpenClaw 的主线。

### 4.1 OpenClaw 的定位

OpenClaw 的母命题可以概括为：

> **一个万能搭子。**

它的核心是：

- 能做事
- 能持续协作
- 能接住多样化任务
- 不是一次性问答，而是长期协作型 agent

### 4.2 ClawPet 的定位

ClawPet 不是另一个独立 AI 产品。

它回答的是另一个问题：

> **这个“万能搭子”应该如何出现在用户身边？**

它的答案是：

- 常驻桌面
- 持续在场
- 可被扫一眼感知
- 关键时刻自然交接
- 逐渐形成关系感

所以最准确的定义是：

> **OpenClaw 是“谁”，ClawPet 是“它如何住进你的桌面”。**

### 4.3 职责边界

**OpenClaw 负责：**

- 核心 agent 能力
- 任务执行
- 多工具协作
- 会话与 run 管理
- 消息、审批、状态源

**ClawPet 负责：**

- 把这些能力变成可感知的桌面存在
- 把关键时刻变成自然交接
- 把原本工具化的体验，转化成一种持续协作关系

换句话说：

> **OpenClaw 提供能力，ClawPet 提供在场感。**

---

## 5. 产品定位

### 5.1 内部一句话定位

ClawPet 是 OpenClaw 的桌面陪伴层：它把 OpenClaw 的 agent 能力转化为可感知、可交接、可陪伴的桌面存在。

### 5.2 对外一句话定位

> **一个住在你桌面上的 AI 搭子：帮你工作，也陪你把高压的一天过完。**

### 5.3 面向官网/传播的版本

> 让 AI 不再躲在终端和面板里。  
> ClawPet 会住在你的桌面边缘，用低打扰但持续在场的方式，告诉你它在做什么、什么时候需要你，并陪你一起把事情做完。

### 5.4 不是什么

ClawPet 不是：

- 传统电子宠物 clone
- 单纯桌面皮 / 通知中心换皮
- 一个与 OpenClaw 深度能力脱钩的普通桌宠
- 泛情感陪聊产品
- 另一个重型 dashboard

## 6. 目标用户

ClawPet 的用户必须区分“切入口”和“长期大盘”。

### 6.1 V1 切入口：OpenClaw / QClaw 重度用户

特征：

- 已有日常 agent 使用习惯
- 会频繁在多个会话间切换
- 需要处理审批、中止 run、补发指令等动作
- 不想一直开完整控制台

他们最在意：

- 能不能一眼知道 agent 状态
- 审批和接球能不能更顺手
- 能不能减少切去 control UI 的次数
- ClawPet 到底是不是“值得长期常驻”

### 6.2 Secondary：AI native 开发者 / 极客

特征：

- 对 AI 工作流和桌面集成极度敏感
- 喜欢 agent 有状态感、人格感、仪式感
- 愿意为了体验质感长期保留一个桌面工具

他们最在意：

- 桌面存在感
- 行为反馈是否自然
- 多 session 是否更易理解
- 产品有没有“新物种感”而不是“更可爱的控件”

### 6.3 长期扩展：高压知识工作者

特征：

- 长时间停留在电脑前工作
- 同时处理多个任务、上下文和中断
- 对“被接住”的感觉有真实需求
- 不一定是 AI power user，但会逐渐依赖 AI 协作

他们最在意：

- AI 是否真的在旁边，而不是埋在后台
- 是否低打扰
- 是否能在忙乱中减少一点认知负担
- 是否兼顾工作价值和情绪缓冲

### 6.4 暂不作为前期主导用户：桌宠爱好者

这类用户未来可能是增量，但不应决定前期方向。

因为他们天然会推动产品走向：

- 收集
- 养成
- 娱乐玩法

这会稀释 ClawPet 最关键的差异化。

## 7. Jobs To Be Done

### 7.1 功能性 JTBD

1. 当我的 agent 在运行时，我想不用盯着控制台也知道它在忙什么。  
2. 当 agent 需要我时，我想及时被接住，但不想被粗暴打断。  
3. 当 agent 完成、失败或卡住时，我想收到清晰、有层次的反馈。  
4. 当我同时跑多个任务时，我想一眼知道哪个最值得我注意。  
5. 当我不想打开完整面板时，我想仍然能完成一些关键操作。  

### 7.2 情绪性 JTBD

1. 我希望我的 AI 助手有存在感，而不是一个躲在后台的黑盒。  
2. 我希望桌面上的 AI 是一个“搭子”，而不是一堆面板。  
3. 我希望高压工作流里有一点温度，而不是纯粹机械化。  

### 7.3 关系性 JTBD

1. 我希望它不像一个随机弹出来的通知，而像一直在旁边的存在。  
2. 我希望它不是“很可爱但没用”，而是“有用，而且让我更愿意一直开着”。  
3. 我希望它逐渐形成一种连续性：它知道我正在工作、正在等待、正在收尾。

## 8. 产品结构

ClawPet 不是单层产品。它应该建立在清晰的分层之上。

### 8.1 Capability Layer（能力层，属于 OpenClaw）

这是 ClawPet 的上游，不由 ClawPet 本身定义。

包括：

- agent 执行能力
- 会话与 run 管理
- 审批请求
- 消息收发
- 工具与状态源

### 8.2 Presence Layer（存在层，属于 ClawPet）

负责把 agent 的运行过程映射成桌面可感知反馈。

包括：

- 状态映射（idle / thinking / executing / waiting / blocked / done / failed）
- 活动映射（read / write / edit / exec / tool / approval 等）
- 动作、速度、情绪、氛围变化
- 多宠物 / 多 session 的空间表达
- glanceable 的状态编码体系

### 8.3 Handoff Layer（交接层，属于 ClawPet）

负责在关键时刻把球递给用户。

包括：

- 审批浮窗
- 完成 / 失败 / 长等待提醒
- 中止 / 快速消息 / 切 session
- 双击进入面板
- 当前最重要任务的轻量入口

### 8.4 Relationship Layer（关系层，属于 ClawPet）

负责让用户从“知道它存在”走向“愿意长期把它留在桌面上”。

包括：

- 气泡语言风格
- 工作状态下的行为习惯
- 生命周期仪式感
- 工作日节奏（开工 / 专注 / 等待 / 收尾）
- 轻量、可信、不过度的陪伴认知

这里要特别强调：

> **Relationship Layer 不是后期锦上添花，而是长期留存逻辑。**

只是它必须建立在真实工作价值之上，而不能先变成空泛情绪产品。

## 9. 核心场景

### 场景 A：后台工作感知

用户正在工作，不想打开面板。  
ClawPet 在桌面边缘用动作、节奏和氛围传达：

- agent 在思考
- agent 在执行
- agent 在等待
- agent 出现异常

目标：**扫一眼就知道大概发生了什么。**

### 场景 B：审批接球

agent 发起高风险执行请求。  
ClawPet 不用冷冰冰弹框，而是用桌面搭子的方式把动作递给用户。

目标：**让审批从“工具中断”变成“自然交接”。**

### 场景 C：完成 / 失败 / 卡住反馈

任务跑完、跑挂、长时间卡住，不再只是状态悄悄变化，而是一个有层次的反馈时刻。

目标：**让 agent 的工作闭环被用户真正接住。**

### 场景 D：轻量工作入口

用户不想每次都打开完整控制台，但仍希望：

- 看状态
- 切 session
- 发一句话
- 中止当前 run
- 处理审批

目标：**宠物是默认入口，面板是轻量扩展，而不是反过来。**

### 场景 E：多 session / 多 agent 协作可视化

用户同时跑多个任务，希望不是在列表里翻，而是能通过不同存在体快速形成空间认知。

目标：**从“一个宠物”升级成“桌面上的工作态势图”。**

### 场景 F：工作日节奏陪伴

一天里，用户会经历：

- 开工
- 长时间专注
- 被等待打断
- 审批 / 决策切换
- 下班前收尾

ClawPet 要在这些时刻成为一种轻量但连续的存在。

目标：**让用户感到“它一直在旁边”，而不是“偶尔跳出来一下”。**

## 10. 产品原则

### 原则 1：先有用，再有爱

没有真实工作价值，陪伴就是空的。

### 原则 2：低打扰，不等于没有存在感

ClawPet 必须安静，但不能安静到像不存在。

### 原则 3：陪伴必须服务工作场景

它不是情绪消费品，而是工作中的关系型产品。

### 原则 4：默认 glanceable

用户主要不是“认真研究它”，而是“余光扫一眼它”。

### 原则 5：先做“值得常驻”，再做“更可爱”

用户愿意一直挂着它，比用户夸它可爱更重要。

### 原则 6：情绪表达必须服务信息传递

动作、语气、颜色和氛围变化，最好都承担某种状态编码或交接意图。

### 原则 7：不伪装情感深度

没有连续性、记忆和可靠行为，就不要装成很懂你。

### 原则 8：以 OpenClaw 为能力边界

如果某项扩展不能显著增强 OpenClaw 的桌面搭子体验，就不应进入前期优先级。

## 11. 当前已有基础

根据现状，ClawPet 已经具备以下原型基础：

- macOS 上独立透明、无边框、常驻桌面的宠物窗口
- 托盘菜单与多宠物管理
- 纯 Canvas 像素渲染
- OpenClaw Gateway 原生接入
- 会话列表与最近对话读取
- 当前 session 消息发送
- run 中止
- exec approval 请求处理
- 审批浮窗
- 灵魂模式 / 文件桥接模式
- 不同状态到宠物行为的初步映射

这说明 ClawPet 已经不是概念验证，而是具备明确产品雏形。

接下来最重要的，不是“再多加点功能”，而是：

> **把它从很酷的原型，推进成一个值得长期常驻的产品。**

## 12. 北极星与成功标准

### 12.1 北极星

> **让用户愿意长期把 ClawPet 挂在桌面上，并真的开始把它当作 OpenClaw 的默认桌面入口。**

### 12.2 v1 要验证的事情

1. 用户会不会长期常驻运行 ClawPet  
2. 用户会不会通过它减少打开 control UI 的次数  
3. 用户会不会通过它完成审批与轻量交接  
4. 用户会不会把它描述为“我的 AI 搭子”，而不是“一个可爱插件”  

### 12.3 关键指标

#### 行为指标

- 日均运行时长
- 常驻开启率
- 通过 ClawPet 完成审批的比例
- 通过 ClawPet 进入 session / 发送消息 / 中止 run 的比例
- control UI 打开频次是否下降

#### 产品健康指标

- Gateway 连接成功率
- 状态同步延迟
- 审批浮窗触达率
- 审批动作成功率
- 原生接入失败后的降级可用率

#### 认知验证指标

- 用户是否会说“它在替我工作”
- 用户是否会说“它一直在旁边”
- 用户是否会说“这是我的 AI 搭子”

## 13. 阶段规划：P0 / P1 / P2

ClawPet 的阶段路线不是简单按“功能多少”划分，而是按**产品价值递进**划分：

- **P0：先让它有资格常驻桌面**
- **P1：再让它像一个真正的桌面搭子**
- **P2：最后让它变成可扩展的 companion system**

---

## 14. P0：桌面常驻资格（Desk-worthy Foundation）

### 14.1 P0 的目标

P0 要回答的问题不是“它够不够酷”，而是：

> **用户有没有理由一直把它挂在桌面上？**

P0 的核心目标：

- 让 ClawPet 可信
- 让 ClawPet 好用
- 让 ClawPet 不吵
- 让 ClawPet 真能承担关键接球

如果 P0 做不好，ClawPet 只会是一个很强的 demo。

### 14.2 P0 的核心价值

完成 P0 后，用户应该形成这几个印象：

- 它能稳定连上 OpenClaw
- 它不是桌面噪音
- 它让我少开很多次控制台
- 当 AI 需要我时，我会相信它递给我的信息

### 14.3 P0 必须包含的内容

#### P0-1 稳定连接与恢复

目标：让 ClawPet 的“在场感”建立在可靠能力之上。

范围：

- Gateway 自动连接与断线重连
- 明确的连接状态枚举（connecting / connected / degraded / disconnected / error）
- 原生接入失败时清晰降级到 soul / file bridge
- 所有关键状态同步给托盘、宠物、面板

#### P0-2 桌面壳与本地持久化

目标：让 ClawPet 像一个会“住下来”的存在，而不是每次重启都重新开始配置。

范围：

- 宠物数量与桌面布局恢复
- 皮肤 / 变体选择恢复
- 点击穿透 / 暂停状态恢复
- 面板最近视图 / 会话偏好恢复
- 启动后保持用户上次的桌面状态

#### P0-3 状态分类与 glanceable 反馈 v1

目标：让用户一眼知道 AI 在做什么类型的工作。

第一版至少支持：

- idle
- thinking
- executing
- approval-requested
- waiting / blocked
- done
- failed
- read / write / edit / exec 等活动差异

要求：

- 不追求极度精细
- 但必须做到“在看 / 在改 / 在跑 / 在等 / 出错”可分辨

#### P0-4 关键交接闭环

目标：让 ClawPet 成为可信的 handoff 入口。

范围：

- 审批浮窗 v2
- 完成反馈
- 失败反馈
- 长时间等待提醒
- 取消 / 中断反馈
- 从提醒快速进入相关上下文

要求：

- 看得懂
- 不焦虑
- 不需要打开控制台才能继续下一步

#### P0-5 轻量操作入口

目标：让用户在不进入完整控制台的情况下完成最常见动作。

范围：

- 查看当前最重要状态
- 快速进入对应 session
- 快速发送一句消息
- 中止当前 run
- 处理审批
- 查看最近结果 / 最近待办接球

#### P0-6 抗打扰机制

目标：让 ClawPet 有存在感，但不会把自己变成桌面噪音。

范围：

- 提醒强度分级
- 同类提醒节流
- 长等待提醒的升级策略
- 点击穿透 / 暂停 / 静音等快速控制
- 对“只是有状态变化，但不值得打断”的情况保持克制

### 14.4 P0 明确不做

- 复杂人格系统
- 多数值养成
- 泛生产力插件扩展
- 宠物资源市场
- 跨平台优先

### 14.5 P0 验收标准

P0 完成时，至少满足：

- 用户重启应用后，桌面状态能恢复  
- OpenClaw 连接异常时有明确降级，不会无声失效  
- 审批与结果接球可以只靠 ClawPet 完成  
- 用户能通过桌宠行为区分至少 5 类关键状态  
- ClawPet 可以被挂一整天，而不让人想关掉  

---

## 15. P1：桌面搭子成形（Companion Credibility）

### 15.1 P1 的目标

P1 要回答的问题是：

> **ClawPet 能不能从“可用工具”升级成“真正的桌面搭子”？**

到了这个阶段，重点不再只是“能不能看状态”，而是：

- 它是不是像一个真的在工作中的存在
- 它是不是能帮助用户建立任务态势感
- 它是不是开始形成低打扰的陪伴感

### 15.2 P1 的核心价值

完成 P1 后，用户应该觉得：

- 它不是静态吉祥物，而是真的在“工作”
- 它的行为是连贯的，不是状态机硬切换
- 多 session 时，它真的能帮我建立空间感
- 它比“更可爱的状态灯”明显更进一步

### 15.3 P1 必须包含的内容

#### P1-1 任务生命周期仪式感

目标：把 agent 工作过程从状态跳变，升级成有节奏的桌面过程。

至少覆盖：

- task received
- thinking
- executing
- waiting
- needs human
- done
- failed

要求：

- 过渡自然
- 状态之间有节奏感
- 用户能感觉到“它经历了一个过程”

#### P1-2 多 session / 多宠物映射

目标：把“多宠物”从装饰性功能变成多任务态势能力。

范围：

- 每只宠物代表一个 session / agent / 工作线程
- 区分主会话与后台会话
- 用存在感、位置、频率、氛围表达优先级
- 支持从宠物进入对应 session

要求：

- 用户不用打开列表，也能大概知道哪个最值得注意
- 不因为宠物变多而信息爆炸

#### P1-3 工作日节奏陪伴

目标：让 Relationship Layer 真正进入产品主线。

范围：

- 开工时的启动感
- 长时间专注时的安静陪伴
- 等待或阻塞时的轻量缓冲
- 完成后的收尾反馈
- 下班前的总结 / 收束感

要求：

- 不要像情感陪聊
- 必须围绕真实工作节奏
- 让用户产生“它一直在旁边”的连续感

#### P1-4 关系语气系统 v1

目标：建立可持续的桌面搭子语言，而不是随机卖萌文案。

范围：

- 气泡语气规范
- 工作状态下的语言分层
- 不同提醒强度的语气差异
- 支持 2-3 种基础 companion 风格预设（不是完整人格系统）

要求：

- 语气可信
- 不幼稚
- 不自作多情
- 长时间看也不腻

#### P1-5 面板 2.0：工作卡片中心

目标：让面板更像桌宠递来的工作卡片，而不是后台控制器。

范围：

- 当前最重要任务
- 待审批 / 待接球动作
- 最近完成 / 最近失败结果
- session 切换与快捷消息
- 降低“管理台感”

#### P1-6 状态语言统一规范

目标：统一动作、颜色、气泡、提醒强度之间的关系。

避免：

- 每个状态各自为战
- 动效、语气、色彩割裂
- 未来新增状态没有边界

### 15.4 P1 明确不做

- 复杂情感记忆
- 泛社交陪伴能力
- 非 OpenClaw 的大范围任务源接入
- 大量自定义编辑器 / 主题市场

### 15.5 P1 验收标准

P1 完成时，至少满足：

- 用户会开始把它描述为“搭子”而不只是“状态灯”  
- 多 session 时，用户能通过 ClawPet 形成态势感  
- 工作流存在更明显的节奏与仪式感  
- 开工 / 专注 / 等待 / 收尾的体验更连续

## 16. P2：可扩展 Companion System

### 16.1 P2 的目标

P2 要回答的问题是：

> **ClawPet 能不能从一个固定实现，成长为一个可扩展的 companion system？**

也就是：

- 不再靠硬编码维持所有表现
- 不同 companion 可以有稳定边界
- 能逐渐形成 pack / schema / 生态能力

### 16.2 P2 的核心价值

完成 P2 后，ClawPet 不只是“当前这只宠物”，而是：

- 一套可扩展的 companion 框架
- 一套可复用的状态 / 行为映射机制
- 一种可成长的桌面 AI 体验层

### 16.3 P2 必须包含的内容

#### P2-1 Companion Schema 外置化

目标：把 companion 的定义从硬编码中抽出来。

范围：

- 宠物资源定义
- 动作定义
- 气泡文案与语气定义
- 状态映射规则
- 事件反应规则

#### P2-2 Personality / Behavior Pack

目标：让不同 companion 有明确差异，但不破坏产品主线。

示例方向：

- 冷静型
- 乐观型
- 黏人但克制型
- 严肃型

要求：

- 只是风格差异，不改变核心工作价值
- 不让产品退化成“换皮市场”

#### P2-3 Skill / Workspace / Role-aware 反应

目标：让 companion 更理解“它正在替你做哪类工作”。

例如：

- 编码任务时有更适合的行为节奏
- 研究 / 搜索任务时有不同反馈逻辑
- 不同 workspace 有不同 companion 气质

#### P2-4 资源与主题体系

长期可形成：

- companion 资源包
- personality pack
- theme pack
- 与 skill / workspace 绑定的角色包

#### P2-5 对更广泛人群的扩展验证

在确保核心不失焦的前提下，验证 ClawPet 是否能从 OpenClaw power user 扩展到更广义的高压知识工作者。

### 16.4 P2 明确不做

- 无边界 pack 商店
- 与核心工作价值无关的大量娱乐资源
- 完全脱离 OpenClaw 的独立 companion app

### 16.5 P2 验收标准

P2 完成时，至少满足：

- 新 companion 行为可在不改核心代码的情况下扩展  
- 风格差异清晰，但主线不失焦  
- 生态扩展不会让产品退化成“桌宠皮肤市场”  

---

## 17. 暂不优先的方向

以下内容不是永远不做，而是当前不应成为主线。

### 17.1 泛生产力插件化扩展

如：

- 天气
- 日历
- 通用提醒
- 番茄钟

除非它们直接增强 OpenClaw 的桌面搭子场景，否则会冲淡主线。

### 17.2 纯娱乐型养成系统

如：

- 喂食
- 抚摸奖励
- 升级数值
- 长期成长线

这些会让项目越来越像桌宠，而不是桌面 companion layer。

### 17.3 泛情感陪聊

如果 ClawPet 没有稳定连续的上下文和真实工作价值支撑，过早强调“深度陪伴”只会显得虚。

### 17.4 跨平台优先

先把 macOS 场景做透，比急着上 Windows / Linux 更重要。

---

## 18. 关键风险

### 风险 1：失焦到普通桌宠

功能越来越多，但和 OpenClaw 工作流关系越来越弱。

### 风险 2：可爱有余，必要性不足

用户觉得它很酷，但并不愿意长期常驻。

### 风险 3：把陪伴讲得太大、做得太空

如果没有真实能力支撑，“陪伴”会变成空口号。

### 风险 4：反馈过度打扰

如果动效、提醒和存在感过强，ClawPet 会从“搭子”变成“桌面噪音”。

### 风险 5：多宠物与多 session 复杂度失控

如果信息层级不清晰，多宠物会从态势图变成认知负担。

---

## 19. 最终产品结论

ClawPet 最值得做大的方向，不是“什么都能做的桌宠”，而是：

> **OpenClaw 最有在场感、最有温度的桌面形态。**

它应该成为桌面上的一个新物种：

- 不是纯 UI
- 不是纯通知
- 不是纯宠物
- 不是纯陪聊
- 而是一个有能力底座、有状态、有交接、有关系感的桌面 AI 搭子

如果这个定义成立，那么产品决策就会简单很多：

优先做：

- 工作流感知
- 关键时刻交接
- 多任务态势可视化
- 工作节奏中的轻量陪伴
- 可信、长期、低打扰的桌面关系感

延后做：

- 泛娱乐
- 泛提醒
- 泛生产力堆料
- 没有能力支撑的空泛陪伴
