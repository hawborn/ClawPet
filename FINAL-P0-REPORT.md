## 🎯 核心功能实现详解

### 1. 长等待提醒升级 (CP-004+) ✅

#### 设计理念
从单点提醒升级为**渐进式多阶段提醒系统**，让用户在长等待时：
1. 首先感受到宠物的"期待"（灵魂状态变化）
2. 然后收到温和的消息提醒
3. 最后升级到系统级别的通知

#### 实现的 5 个阶段

```typescript
30s   → "期待" 阶段（灵魂状态更新，宠物行为变化）
60s   → "第一次提醒"（温和的消息）
120s  → "第二次提醒"（再次提醒）
300s  → "重要提醒"（Dock 弹动 + 提醒）
600s  → "系统通知"（强烈提醒 + 窗口聚焦）
```

#### 关键特性
- **独立定时器追踪**：每个任务的 5 个阶段都独立管理
- **智能节流**：提醒强度越高，节流时间越宽松
  - hint1/hint2: 30s 内不重复
  - important: 30s 内不重复
  - critical: 不节流（确保用户一定能看到）
- **自动清理**：任务完成/失败时自动清除所有定时器

#### 代码架构

```typescript
// 核心函数三角形
setupWaitingStageTimers(runId)      // 建立定时器
  ↓
handleWaitingStage(runId, stage)    // 处理各阶段
  ↓
clearWaitingStageTimers(runId)      // 清理定时器
```

### 2. 宠物行为优化 (CP-006) ✅

#### Waiting 状态行为升级

**原设计**（不理想）
```
65% idle（无聊的等待）
35% greet（随意的问候）
```

**新设计**（优化后）
```
50% greet（积极的聆听）← +15%
35% idle（专注的等待）  ← 保持
15% walk（焦急的踱步）  ← 新增
```

**改进效果**：
- 宠物看起来更"在乎"用户的任务
- 踱步动作表现出焦急感
- 整体更像一个"有感情的助手"

## 🎓 技术亮点

### 1. 多级定时器管理系统
- 使用 Map 而不是单个变量，支持并发任务
- runId-stage 复合键便于追踪
- 自动清理防止内存泄漏
- 易于扩展更多阶段

### 2. 渐进式提醒策略
- 从不可见 → 可见 → 重要 → 系统
- 每个阶段有明确的用户体验目标
- 与现有的 4 级提醒系统无缝整合
- 节流机制防止提醒疲劳

### 3. 有机的行为映射
- 灵魂状态 → 宠物行为的自然映射
- 概率组合产生多样化行为
- 状态变化时自动更新
- 易于调整和优化

## 💡 关键决策说明

### 为什么选择 5 个阶段？
- 30-60-120：快速反馈层（针对短等待）
- 300-600：重要警告层（针对长等待）
- 覆盖"忘记了"到"真的遗忘"的全段

### 为什么优化 waiting 状态？
- 用户最关心的状态（频繁出现）
- 视觉反馈最丰富（看起来最"有感"）
- 改进空间最大（从 2 种行为到 3 种）

### 为什么用灵魂状态而不是直接改活动？
- 保持架构一致性（灵魂状态是逻辑层）
- 支持多宠物同步（宠物管理器统一处理）
- 易于未来扩展（如不同灵魂的不同行为映射）
